## 概要
2025〜2026年時点で「公式に公開された“気象庁REST API”」という形の包括的な新API／新エンドポイントの情報は見当たりませんでした。  
一方で、気象庁は防災情報の**オープンデータ化・機械判読形式（XML/JSON 等）での配信**を拡充しており、2023〜2024年にかけて新形式や英語情報の整備が進んでいます。  
多言語対応の災害情報システムでは、「気象庁HP・防災情報XML（電文）・自治体／民間API」の組み合わせと、変更検知の仕組みを実装するのが現実的です。

- 2024年までの気象庁公式情報・オープンデータ方針  
- 周辺省庁・白書等における「気象庁オンライン情報サービス」への言及[3]  
に基づく整理と、そこからの**実務的な設計指針（推奨アーキテクチャ）**になります。

## 主要な発見・ポイント
- **2025–2026年時点で「気象庁公式REST API」としての新エンドポイント公開情報は確認できない**
- 気象庁の機械判読データは依然として**防災情報XML（電文）・各種オンラインサービス**が中心（JSONネイティブAPIは限定的）
- 環境省等の資料で「気象庁オンライン気象情報サービス」が**電力・再エネ分野向けに利用されていることが明記**されている[3]
- 中長期のデジタル政策・防災DXの文脈で、**防災情報のオープンデータ化・標準化**が促進されている[3][4]
- 英語を含む**多言語の防災情報ページ／ガイドライン**は拡充傾向にあり、多言語UI構築の参考になる
- P2P地震情報APIなど**民間／コミュニティAPI**が、実質的な「JSON APIレイヤー」として利用されている
- 変更点把握には、**気象庁サイトの更新情報RSS・防災情報XML仕様書・GitHubミラー**を継続的にモニタリングする必要
- 今後の拡張を見越し、**疎結合なアダプタ層（気象庁XML／民間APIなどソースごとに分離）**で実装するのが安全
- 多言語対応のため、**元データは日本語のみという前提で翻訳レイヤー（用語辞書＋機械翻訳）を独自に持つ**必要がある
- 将来的な公式API化に備え、**スキーマバージョン管理とテスト（サンプル電文による回帰テスト）**を用意しておくと移行が容易

## 詳細情報

### 1. 気象庁APIの現状整理（2024年まで）

1. **「API」という名称の公式サービス**
   - 気象庁は、一般開発者向けの明確な「REST API ドキュメントサイト」を公開していません。
   - 代わりに、以下が実質的な「APIソース」として使われています（名称は便宜的）。
     - 防災情報XML（気象庁防災情報XML フォーマット／電文）
     - 気象庁ホームページの各種JSON/CSV（レーダー・アメダス・台風等、一部でJSONが利用されている例もある）
     - FTP/HTTPで提供される数値予報データ等（研究用など）

2. **環境省資料に見られる位置づけ**
   - 温室効果ガス・再エネ系の資料で、**「気象庁のオンライン気象情報サービス」**がシステム内で利用されていることが明記されています[3]。
   - 表2-4「気象庁のオンライン気象情報サービス」では、気象庁がオンラインで機械的に利用できる情報サービスを提供している事実が示されています[3]。
   - 同資料では、**MSM（メソ数値予報モデル）を利用した太陽光発電出力予測システム**について、気象庁データを用いた検証や誤差評価が記載されています[3]。
   - これは、気象庁が外部システムからの自動取得を前提としたデータ提供を継続していることを示します。

3. **防災情報XML（電文）の継続性**
   - 地震・津波・気象警報などは、防災情報XML（電文）として提供されており、多数のシステムがこれを機械読取して利用しています。
   - フォーマット仕様の改定は行われるものの、**XML 電文の廃止・全面JSON化といった動きは確認されていません**（2024年時点までの情報）。

### 2. 2025〜2026年の公式情報の空白と、その意味

- 検索可能な範囲では、2025〜2026年にかけての「気象庁 新API」「新RESTエンドポイント」に関する**公式発表・仕様書**は見つかっていません。
- 一方で、国全体としてのデジタル・オープンデータ方針は、以下の点を明確にしています。
  - デジタル社会形成基本法に基づき、行政情報を**機械判読可能な形式で公開**すること[4]。
  - 防災を含む公共分野で**民間利活用を促進**する方針[4][5]。
- これらから、気象庁についても
  - 既存のオンライン気象情報サービス／防災情報XMLの**安定運用・拡充**
  - 将来的なAPI化・標準化への**漸進的な移行**
  を継続していると考えられます（ただし、具体的な新エンドポイントは現時点で確認不能）。

### 3. P2P地震情報API と民間APIレイヤーの役割

- P2P地震情報APIは、気象庁の地震情報をもとに**コミュニティが独自整形したJSON APIレイヤー**です。
- このような民間APIは、
  - **JSON形式・CORS対応・安定したスキーマ**の提供
  - 気象庁の仕様変更に応じた**吸収レイヤー**
  の役割を持ちます。
- 多言語対応の災害情報システムでは、
  - 「一次ソース：気象庁XML/オンラインサービス」
  - 「二次ソース：P2P地震情報API や他社API」
  の両方を利用し、**冗長化とフォールバック**を設計しておくと安定します。

### 4. 多言語対応に関する最新トレンド（2023–2025）

- 国・自治体レベルで、防災情報の多言語化が急速に進んでいます（やさしい日本語／英語／中国語など）。
- 気象庁自身も、  
  - 英語の解説ページや防災情報を提供  
  - 国際向けの情報提供（航空気象など）で英語を標準利用  
  しています。
- ただし、「多言語JSON API」として公開されているわけではなく、**UI向けHTMLやPDF／説明文**が中心です。
- よって、多言語対応システムでは
  - **日本語電文 → 構造化パース → 項目ごとの翻訳**
  - 固有名詞（地名・気象用語）は**対訳辞書ベース**で翻訳
  - 定型文は**テンプレート＋ローカライズファイル（i18n）**で管理
  といったアプローチが必要になります。

### 5. 変更点の検知・追従のための運用

2025〜2026年に新エンドポイントが追加された場合、**最初に気付くための仕組み**が重要です。

- 監視すべき情報源（推奨）
  - 気象庁トップページの「お知らせ」「報道発表」「技術資料」
  - 防災情報XMLの仕様書・更新履歴ページ
  - 関連省庁の白書／中長期計画（特に防災・データ・デジタル関連）[3][4][5]
- 技術的な変更検知
  - 既存XML・JSONの**スキーマ差分チェック**を定期的に実行
  - サンプル電文・過去ログに対して**回帰テスト**を走らせる
  - 想定外フィールド／欠落時にアラートを上げる仕組み
- コミュニティ情報
  - P2P地震情報や防災系OSSプロジェクトの更新（GitHubリポジトリなど）も、新仕様の早期情報源になりやすい

## 実装への示唆

ここからは、実際に「多言語対応の災害情報提供システム」を設計・実装するうえでの**具体的なアーキテクチャ提案**です。

### 1. アーキテクチャ全体像

- **データ取得層**
  - アダプタA：気象庁 防災情報XML（HTTP/FTP等）
  - アダプタB：気象庁オンライン気象情報サービス（CSV/JSON相当）[3]
  - アダプタC：P2P地震情報API（JSON）
  - アダプタD：その他民間API（レーダー、雨雲、河川水位など）
- **正規化・統合層**
  - 各アダプタからのデータを、内部モデル（例：QuakeEvent, WeatherWarning, TsunamiAlertなど）にマッピング
  - フィールドにバージョン情報を付与（`schemaVersion`）
- **多言語化層**
  - 固有名詞辞書（地名／気象用語）: ja → en/zh/…  
  - テンプレートメッセージ: ICU MessageFormat 等で管理
  - 動的部分（日時・マグニチュード・階級など）をロケールに応じてフォーマット
- **配信層**
  - REST/GraphQL API（自システムの公開API）
  - WebSocket/SSE でのリアルタイム配信
  - LINE / Telegram / WebPush 等のチャネル連携

### 4. 変更耐性のある設計

- **スキーマバージョンの付与**
  - `jmaXmlVersion`, `p2pApiVersion` といったメタ情報をイベントに付加。
- **フェイルソフト**
  - 未知フィールドはログに残しつつ無視する。
  - 重要フィールド欠落時は「簡易メッセージ（最低限の情報だけ）」にフォールバック。
- **テスト戦略**
  - 代表的なXML電文・JSONレスポンスを収集して「ゴールデンファイル」とし、
  - パーサー・マッピングロジックに対して**スナップショットテスト**を実施。

### 5. 今後の公式API化への備え

- 新たな「気象庁REST API」が正式公開された場合に備え、
  - アダプタ層に**新しい JmaRestAdapter**を追加できるようインターフェースを定義しておく。
  - 内部モデルへのマッピングは共通化しておく。
- APIが登場すれば、**XML → JSON の変換ロジックの多くを廃止**できるため、  
  アダプタを差し替えるだけで移行できる構造にするのが有利です。
